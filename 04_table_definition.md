# 一つ目のER図レビュー
 
## 全体
-テーブル名の先頭は小文字にしてください。データベースの命名規則として原則大文字は利用しないです。
-複数単語の場合、命名規則としてスネークケースを利用します。Cart item,Buy detailsテーブルの名前がスネークケイースで記述できていません。
  - [check]とても細かくて恐縮ですが、「スネークケイース」→「スネークケース」です。
  
- [add] "created_at"、"updated_at"が同じ名前になっています。指摘文を考え、追記してください。
- [add] "id"は全てのテーブルに必要ですが、準備されていますか。個別に指摘していただいている箇所もありますが、全体の指摘事項として、追記してください。

## admin
-admin_id,admin_email,admin_passwordについてです。テーブル名はカラム名の頭につけないです。テーブル名がadminとなっているので、テーブル名をつける必要がありません。
- [add]PKがありません。指摘文を考え、追記してください。

## users
-PKとなるカラム名はidです。
- [add]配送先が一つしか登録できないようですが、複数登録できるようにする必要があります。
  本来であればER図での指摘事項ではありますが、もし足りていない場合こちらで指摘が必要です。

-products・discs・labels・artists・genre・cart_itemテーブルについてです。products_idなのですが、外部キーはテーブル名単数形_idとなります。
  - [check]
    - こちらの情報は"products"テーブルの指摘事項として盛り込みましょう。
    - また、productsテーブルの"products_id"はPKなのに対し、他テーブルはFKとなっています。
      「discs,labels,artists,genre,cart_itemテーブルについても、命名規則は同様です」と明記すると、修正すべきポイントがわかりやすくなるかと思います。
  
-songs,labels,artists,genre,cart_itemテーブルについてです。PKがありません。レコードを一意に識別するために必要です。
  - [check]各テーブルで指摘するか、全体の指摘事項としてまとめると分かりやすいです。

## songs 
-FKにAUTO INCREMENTがついていますが、一つのディスクに複数の曲名が保存できるでしょうか。
  - [check]カラム名から察するに、PKのつもりで付けたのでしょう。ですので、指摘としては「disc_idをPKとして、カラム名を見直す」というのが適切になります。
- [add]"song"は曲名を持たせるカラムとなっていますが、"name"や"title"など、名前であることを一目見て分かるような命名にする必要があります。
  これは、"label", "artists", "genre"の各テーブルでも同様のことが言えます。
- [add]"song"カラムは曲名を表しますが、integer型となっております。適切な型が何であるか、問題提起するレビュー文を追記しましょう。
  ("artists", "genre"テーブルにも同様のケースが存在します。)

## artists
-FKにAUTO INCREMENTがついていますが、一つの商品には複数の曲が保存されています。歌う人が異なる場合にはデータにどのように保存を行うのでしょうか。
 - [check]songs同様に、PKのつもりで付けたのでしょう。ですので、指摘としては「disc_idをPKとして、カラム名を見直す」というのが適切になります。

## buy_details
-命名規則として、複数単語のカラム名はスネークケースにします。
- [add]命名規則として、"buy_details"ですと「購入詳細」という意味となり、エンドユーザー側の視点のみとなってしまいます。
  "order_details"など、注文であることが分かる = 管理者から見た時の役割も明確であるような命名規則にしてもらうよう、レビュー文を追加してください。
- [add]"buy_num"は直訳すると「番号を買う」となり、購入枚数とはやや意味合いが異なってきます。適切な命名にしてもらうよう、レビュー文を追加してください。

## buy 
-buy_details_idについてです。上記と重複してしまうのですが、外部キーはテーブル名単数形_idとなります。
  - [add]リレーションを考えた時、そもそも"buy_details_id"は必要でしょうか。
    本来であればER図で指摘するポイントですが、テーブル定義書に記載してしまうケースもあります。ここでもチェックできるようにしておきましょう
- [add]命名規則として、"buy"ですと「購入する」という意味となり、エンドユーザー側の視点のみとなってしまいます。
  "orders"など、注文であることが分かる = 管理者から見た時の役割も明確であるような命名規則にしてもらうよう、レビュー文を追加してください。
- [add]支払い方法のカラムがpayとなっておりますが、支払い金額なのか、支払い方法なのか、一見すると迷ってしまいます。
  支払い方法と分かるような命名規則にしてもらうよう、レビュー文を追加してください。
- [add]支払い方法のデータ型がstringとなっていますが、enumで実装を勧めたいところです。レビュー文を追加しましょう。
- [add]支払い合計のカラム名が"stock"となっていますが、合計を表す単語としてはやや意味合いが異なっていますので、別名にしてもらうようレビュー文を追加してください。
- [add]送付先住所カラムがありますが、郵便番号や宛先氏名のカラムも必要です。

## cart_item
-中間テーブル名は複数＿複数となります。
  - [check]cart_itemは中間テーブルではなく、カート内商品を保持しておくためのカラムですので、"cart_items"とすれば問題ありません。
- [add]"buy num"は直訳すると「番号を買う」「購入番号」となり、カラム説明と意味が異なってきます。適切な単語を選んでください。
- [add]"subtotal"を小計金額としてカラムで持っていますが、不要です。不要な理由を考え、レビュー文に追記してください。

## discs
 - track_noはディスクではなく、曲順を管理するために使う属性ですので、このテーブルに持たせるのは不適切です。
 - 現在のカラム構成ですと、同一商品内で異なるディスクであるかの判断がつきません。ディスクNo、ディスク名などの判断材料が必要となります。
